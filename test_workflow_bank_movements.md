# Bank Account — Full Manual Frontend Test Checklist

Run **every item** on **both frontends**: `legacy_frontend/` and `new_frontend/`.

---

## Phase 1: Bank Account CRUD (API: `bank_account.php`)

- [ ] **1.1** Open Bank Accounts list screen. Page loads without errors.
- [ ] **1.2** CREATE: Fill all fields (name, alias, bank, account number, currency, email). Submit.
- [ ] **1.3** Verify the new account appears in the list immediately.
- [ ] **1.4** CREATE with minimum fields only (name, bank, account). Submit. Should succeed.
- [ ] **1.5** CREATE with empty name — should show validation error on the UI, not a crash/500.
- [ ] **1.6** CREATE with empty account number — validation error?
- [ ] **1.7** EDIT: Change name, alias, or bank. Save. Reload the list. Changes persist.
- [ ] **1.8** EDIT: Change currency. Save. Verify change appears.
- [ ] **1.9** DELETE / DEACTIVATE: Deactivate an account. It should disappear from the active list or show as inactive.
- [ ] **1.10** Try deleting an account that has movements linked — is it blocked with a message? Does it silently succeed? (This is a potential bug spot.)

---

## Phase 2: Assign Bank Account to Store (API: `store_bank_account.php`)

- [ ] **2.1** Open Store → Bank Accounts / Payment Methods screen.
- [ ] **2.2** List existing store-bank-account assignments. Page loads.
- [ ] **2.3** CREATE: Pick a store, a bank account, give it a name, choose default transaction type (CASH). Save.
- [ ] **2.4** Verify the assignment appears in the list.
- [ ] **2.5** CREATE: Same store + same bank account but different transaction type (CREDIT_CARD). Does the UI allow duplicates? What message?
- [ ] **2.6** EDIT: Change the `default_transaction_type` on an existing assignment. Save. Verify.
- [ ] **2.7** DELETE: Remove/Deactivate an assignment. Verify it disappears.
- [ ] **2.8** CREATE with no transaction type selected — what happens?
- [ ] **2.9** CREATE with no bank account selected — what happens?

---

## Phase 3: Sale — CASH (API: `order_info.php` + `payment_info.php`)

- [ ] **3.1** Open POS / Sell screen.
- [ ] **3.2** Add one or more items to the cart. Verify totals calculate correctly.
- [ ] **3.3** Complete the sale using **CASH** as the payment method.
- [ ] **3.4** Verify the order appears in the Orders list with correct total.
- [ ] **3.5** Verify the order `paid_status` shows as **PAID**.
- [ ] **3.6** Open Bank Movements screen. Find the movement generated by this sale.
- [ ] **3.7** Verify the bank movement fields:
  - `transaction_type` = **CASH**
  - `type` = **income**
  - `total` matches the sale total
  - `amount_received` matches the sale total (or what was tendered)
  - `bank_account_id` matches the store's linked bank account
  - `balance` is populated (not NULL, not 0)
  - `paid_date` is set
- [ ] **3.8** Verify the bank movement is linked to the correct order (expand/view Bank Movement Orders).
- [ ] **3.9** Open the Bank Account detail/movements view. Verify the movement appears under that account.

---

## Phase 4: Sale — All Payment Methods

Repeat Phase 3 for each type. Each one must produce a bank movement with the correct `transaction_type` and `bank_account_id`:

- [ ] **4.1** **CREDIT_CARD** — set up store_bank_account with `default_transaction_type = CREDIT_CARD` first. Do the sale. Verify bank movement.
- [ ] **4.2** **DEBIT_CARD** — same setup with `DEBIT_CARD`. Do the sale. Verify bank movement.
- [ ] **4.3** **TRANSFER** — same setup with `TRANSFER`. Do the sale. Verify bank movement.
- [ ] **4.4** (If DIGITAL_WALLET exists in your system) — test it too.

---

## Phase 5: Sale — Multiple Payment Methods in ONE Payment

- [ ] **5.1** On a single order, split payment: e.g., 60% CASH + 40% CREDIT_CARD in the **same** payment transaction.
- [ ] **5.2** Verify **two** bank_movement records were created for that one payment.
- [ ] **5.3** Verify each bank movement has its own `transaction_type` and `bank_account_id`.
- [ ] **5.4** Verify each bank movement links to the same order (via bank_movement_order).

---

## Phase 6: Partial Payments on One Order

- [ ] **6.1** Create an order.
- [ ] **6.2** Make a **partial payment** (e.g., 50% of total in CASH).
- [ ] **6.3** Verify order `paid_status` = **PARTIALLY_PAID**.
- [ ] **6.4** Verify bank movement shows only the partial amount.
- [ ] **6.5** Make a **second payment** for the remaining 50% (e.g., in CREDIT_CARD).
- [ ] **6.6** Verify order `paid_status` = **PAID**.
- [ ] **6.7** Verify **two** bank movements exist, with correct amounts and transaction types.
- [ ] **6.8** Open the Order Detail screen. Verify the payment history / applied payments list shows both payments with correct amounts and methods.

---

## Phase 7: Bank Account Balance — Running Balance Validation

- [ ] **7.1** Pick a bank account that now has 3+ movements from the tests above.
- [ ] **7.2** Open the bank account detail / movement list view.
- [ ] **7.3** Verify the **running balance** column increments correctly:
  - Movement 1 (income): `balance` ≈ `total`.
  - Movement 2 (income): `balance` = previous balance + new total.
  - Movement 3 (income): `balance` = previous balance + new total.
- [ ] **7.4** If any movement is an **expense**, verify `balance` **decreases**: `balance` = previous balance − total.
- [ ] **7.5** Verify the balance displayed on the bank account summary/card matches the latest movement's `balance`.

---

## Phase 8: Purchase Flow (API: `purchase_info.php` + `bill_info.php`)

- [ ] **8.1** Open Purchases / Compras screen.
- [ ] **8.2** CREATE a purchase: select provider, add items with quantities and unit prices. Save.
- [ ] **8.3** Verify the purchase appears in the purchase list with correct total.
- [ ] **8.4** Create a **bill** from the purchase (or linked to the purchase). Fill in folio, due date, bank account to pay from.
- [ ] **8.5** Verify the bill appears with `paid_status = PENDING` or similar.
- [ ] **8.6** **Pay the bill**: select a bank account and payment method. Submit.
- [ ] **8.7** Verify bill `paid_status` changes to **PAID**.
- [ ] **8.8** Open Bank Movements. Find the movement for this bill payment.
- [ ] **8.9** Verify bank movement:
  - `type` = **expense**
  - `transaction_type` matches what was selected
  - `total` matches the bill payment amount
  - `bank_account_id` is correct
  - `balance` decreased by the payment amount
- [ ] **8.10** Verify a **bank_movement_bill** link exists (may be visible in bill detail or bank movement detail).
- [ ] **8.11** Repeat with a **partial** bill payment. Verify bill status. Then pay remaining. Verify full paid.

---

## Phase 9: Direct Expense (API: `expense_info.php`)

- [ ] **9.1** Open Expenses / Gastos screen.
- [ ] **9.2** CREATE a direct expense (not from a purchase — e.g., rent, utilities). Fill concept, amount, bank account, payment method. Save.
- [ ] **9.3** Verify the expense appears in the expense/payment list.
- [ ] **9.4** Open Bank Movements. Find the expense movement.
- [ ] **9.5** Verify:
  - `type` = **expense**
  - `total` matches the expense amount
  - `bank_account_id` is correct
  - `balance` decreased
- [ ] **9.6** If the expense was linked to an order (via `bank_movement_order`), verify the order's `amount_paid` updated.

---

## Phase 10: Cancel a Payment (API: `updates/cancel_payment.php`)

- [ ] **10.1** Pay an order (creates bank movement). Note the order state and bank movement.
- [ ] **10.2** Open the payment detail. Click **Cancel Payment**.
- [ ] **10.3** Verify payment `status` shows as **DELETED** (or similar cancelled state).
- [ ] **10.4** Verify order `paid_status` rolls back correctly (from PAID → PENDING, or from PARTIALLY_PAID → PENDING).
- [ ] **10.5** Check bank movements — does the movement still exist? Does its status change? (This is a known potential discrepancy area.)
- [ ] **10.6** Check bank account balance — does it recalculate after cancellation? (Known issue: balance may not auto-correct.)

---

## Phase 11: Cancel a Bill Payment

- [ ] **11.1** Pay a bill (creates bank movement, expense-type).
- [ ] **11.2** Cancel that payment from the bill detail.
- [ ] **11.3** Verify bill `paid_status` rolls back.
- [ ] **11.4** Check bank movement status and bank account balance.

---

## Phase 12: Bank Movement Checkpoint (API: `updates/set_bank_movement_checkpoint.php`)

- [ ] **12.1** Pick a bank movement that has a verified correct balance.
- [ ] **12.2** Set it as a **checkpoint** (if the UI exposes this — edit bank movement, flag as checkpoint, set the trusted balance).
- [ ] **12.3** Verify `is_checkpoint` is marked on that movement.
- [ ] **12.4** Create a new sale/payment (new bank movement after the checkpoint).
- [ ] **12.5** Verify the new movement's `balance` starts from the checkpoint's balance, not from 0.
- [ ] **12.6** If the UI allows, **change** the checkpoint to a different movement. Old checkpoint should be cleared, new one active.

---

- [ ] **13.4** Verify the balance did **not** change (amount is the same, only type changed).

Run steps 1–16 identically on:

- [ ] **17.1** Legacy Frontend (`legacy_frontend/` — POSReservaciones)
- [ ] **17.2** New Frontend (`new_frontend/` — POSProcesos)

For the **same data** on both:
- [ ] Bank movement list shows same movements, same balances, same transaction types.
- [ ] Bank account balance number is identical.
- [ ] Order paid status is identical.
- [ ] Bill paid status is identical.

## Phase 18: Edge Cases & Bug Hunting

- [ ] **18.1** Bank movement with **no** `bank_account_id` (cash-only movement): should appear without a balance or with `balance = NULL`.
- [ ] **18.2** Sale on a store with **no** `store_bank_account` for that transaction type: what happens? Graceful error? Falls back to some default?
- [ ] **18.3** Sale on a store with **multiple** `store_bank_account` for the same `default_transaction_type`: which one is used?
- [ ] **18.4** Attempt to **delete** a bank account that has linked movements: blocked? Allowed? What message?
- [ ] **18.5** Create a sale where `received_amount > payment_amount` (overpayment with change): verify `change_amount` is correct, bank movement reflects `received_amount` not `payment_amount`.
- [ ] **18.6** Create a sale in a **non-default currency** (if supported): verify `exchange_rate` and `currency_id` on bank movement.
- [ ] **18.7** Very large amounts (e.g., 999,999.99): any UI truncation or rounding errors?
- [ ] **18.8** Negative amounts (if allowed for adjustments): does the UI handle it?
- [ ] **18.9** Rapid consecutive sales: do bank movements always get assigned the correct `bank_account_id`? No race condition where a movement gets NULL `bank_account_id`?
